# EasyFinance — Анализ_GAP между сайтом, API и Flutter-приложением

**Дата составления:** 5 июля 2026 г.  
**Версия документа:** 1.0  
**Цель:** Выявление расхождений между обещаниями на сайте EasyFinance, текущим состоянием API и реализацией в Flutter-приложении, формулирование приоритизированных запросов к разработчику API.

---

## 1. Введение

Проект EasyFinance включает три ключевых компонента:

1. **Сайт (easyfinance.ru)** — публичная презентация продукта с перечнем функциональных возможностей, обещанных пользователям.
2. **API (EF_API.md)** — серверная часть, предоставляющая интерфейсы для работы с данными пользователей.
3. **Flutter-приложение** — клиентская часть, реализующая пользовательский интерфейс и бизнес-логику на стороне клиента.

Настоящий документ проводит **комплексный GAP-анализ** между этими тремя слоями и выявляет:

- Какие функции сайта **полностью поддерживаются** API;
- Какие функции сайта поддерживаются **частично** или с ограничениями;
- Какие функции сайта **вообще не имеют** серверной реализации;
- Какие ограничения API **тормозят** развитие Flutter-приложения.

Результаты анализа оформлены в виде приоритизированного списка запросов для передачи разработчику API.

- ✅ **Есть** — функция полностью поддерживается сервером и клиентом
- ⚠️ **Частично** — есть ограничения, требующие доработки
- ❌ **Нет** — функция отсутствует на серверном уровне

## 3. Детальный разбор пропущенного функционала

### 3.1. Календарь плановых/повторяющихся платежей

**Что обещает сайт:** Календарь с плановыми и повторяющимися платежами, напоминания по SMS/email, синхронизация с Google Calendar.

**Что есть в API:** Ничего. Операции создаются только вручную через operations.post. Нет concept для recurring operations.

**Что нужно добавить:**
- Эндпоинт `operations.recurring.post` — создание шаблона повторяющейся операции (частота: ежедневно, еженедельно, ежемесячно, ежегодно, произвольный cron).
- Эндпоинт `operations.recurring.get` — список активных повторяющихся операций.
- Эндпоинт `operations.recurring.set` — изменение/деактивация шаблона.
- Параметр `recurrence_id` в operations.get — для привязки операции к шаблону.
- Серверная генерация будущих операций (scheduler на бэкенде) или клиентская генерация при открытии календаря.

**Приоритет:** 🔴 Высокий — одна из ключевых функций сайта.

---

### 3.2. Уведомления и напоминания (SMS / Email / Push)

**Что обещает сайт:** Напоминания о плановых платежах по SMS и email, интеграция с Google Calendar.

**Что есть в API:** Отсутствует полностью.

**Что нужно добавить:**
- Эндпоинт `notifications.settings.get` — получение настроек уведомлений пользователя.
- Эндпоинт `notifications.settings.set` — настройка каналов (SMS, email, push), времени напоминаний.
- Эндпоинт `notifications.history.get` — история отправленных уведомлений.
- Сервис отправки уведомлений на бэкенде (интеграция с SMS-провайдером, email-сервисом, FCM/APNs для push).
- Webhook / push-уведомления для Flutter-приложения.

**Приоритет:** 🔴 Высокий — напоминания критичны для удержания пользователей.

---

### 3.3. EasyBank — банковская синхронизация

**Что обещает сайт:** Импорт банковских карт из 200+ банков в 15 странах.

**Что есть в API:** Отсутствует полностью.

**Что нужно добавить:**
- Интеграция с агрегаторами банковских данных (Plaid, Tinkoff Open API, Sinopse, Salt Edge или аналоги).
- Эндпоинт `banks.connect` — инициация подключения банковского счёта (OAuth-поток).
- Эндпоинт `banks.accounts.get` — список подключённых банковских счетов.
- Эндпоинт `banks.sync` — запуск синхронизации транзакций.
- Эндпоинт `banks.status` — статус подключения и последней синхронизации.
- Периодическая фоновая синхронизация транзакций (cron/webhook).

**Приоритет:** 🟡 Средний — требует серьёзных инвестиций и партнёрских соглашений; может быть отложена.

---

### 3.4. Бюджет — детализация по категориям

**Что обещает сайт:** Бюджет планирования для доходов/расходов/кредитов/депозитов с прогнозами.

**Что есть в API:**
- `budget.get` возвращает **только агрегат** `{planned: number, spent: number}` — без разбивки по категориям.
- Создание/обновление бюджета возможно только через `users.post` (не через standalone `budget.post/set`).

**Что нужно добавить:**
- Эндпоинт `budget.categories.get` — получение плановых значений по каждой категории.
- Эндпоинт `budget.categories.post` — установка плана по конкретной категории.
- Эндпоинт `budget.categories.set` — изменение плана по категории.
- Эндпоинт `budget.forecast` — серверный прогноз расходов на основе истории.
- Добавить `budget.post` и `budget.set` как самостоятельные эндпоинты (вместо вложенности в users).

**Приоритет:** 🔴 Высокий — без категорий бюджета невозможно реализовать тахометр и аналитику.

---

### 3.5. Тахометр расхода бюджета

**Что обещает сайт:** Визуальный индикатор скорости расхода бюджета (тахометр).

**Что есть в API:** Зависит от данных бюджета по категориям (отсутствует — см. п. 3.4).

**Что нужно добавить:**
- Рассчитывается на основе данных `budget.categories.get` и `operations.get` за период.
- Серверный эндпоинт `budget.speedometer` или клиентский расчёт на основе доступных данных.
- Параметры: категория, период (день/неделя/месяц), скорость расхода (₽/день).

**Приоритет:** 🟡 Средний — зависит от реализации п. 3.4.

---

### 3.6. Графики и аналитика

**Что обещает сайт:** Графики и аналитика расходов/доходов.

**Что есть в API:** Отсутствуют специализированные эндпоинты. Всё приходится считать на клиенте.

**Что нужно добавить:**
- Эндпоинт `analytics.summary` — сводка по периоду (доходы, расходы, баланс, топ категорий).
- Эндпоинт `analytics.trend` — тренд по месяцам/неделям.
- Эндпоинт `analytics.category` — разбивка по категориям за период.
- Эндпоинт `analytics.merchant` — аналитика по продавцам/MCC-кодам.
- Параметры фильтрации: период, категории, валюты, теги.

**Приоритет:** 🟡 Средний — возможно рассчитывать на клиенте, но серверные эндпоинты оптимизируют нагрузку и позволяют делать сложные запросы.

---

### 3.7. Рекомендации (тахометры рекомендаций)

**Что обещает сайт:** Персонализированные финансовые рекомендации.

**Что есть в API:** Отсутствуют эндпоинты.

**Что нужно добавить:**
- Эндпоинт `recommendations.get` — список рекомендаций на основе анализа операций пользователя.
- Алгоритмы: «вы тратите больше в категории X, чем в среднем», «стоит перевести депозит на более выгодную ставку» и т.д.
- Эндпоинт `recommendations.dismiss` — скрытие рекомендации.

**Приоритет:** 🟢 Низкий — может работать на клиенте на основе локальных данных.

---

### 3.8. Кастомные категории — стабильность

**Что обещает сайт:** Создание пользовательских категорий.

**Что есть в API:** `categories.post` и `categories.set` **документированы**, но могут **не работать** для не-партнёрских приложений.

**Что нужно добавить:**
- Гарантировать работоспособность `categories.post` и `categories.set` для сторонних приложений.
- Добавить параметр `app_id` или `partner_key` для авторизации.
- Или: выделить публичный API-слой для сторонних разработчиков.

**Приоритет:** 🔴 Высокий — без кастомных категорий приложение ограничено в функциональности.

---

### 3.9. Теги — запись и управление

**Что обещает сайт:** Теги на операциях.

**Что есть в API:** `tags.get` — чтение тегов. **Нет** `tags.post` / `tags.set` — нельзя создавать и редактировать теги.

**Что нужно добавить:**
- Эндпоинт `tags.post` — создание нового тега.
- Эндпоинт `tags.set` — переименование/удаление тега.
- Эндпоинт `tags.delete` — удаление тега.
- Привязка тегов к операциям через `operations.post` (уже поддерживается через параметр `tags`).

**Приоритет:** 🟡 Средний — теги уже можно использовать через `tags.get` и отправлять в операциях, но создание новых тегов невозможно.

---

### 3.10. Кредиты и депозиты — выделенные эндпоинты

**Что обещает сайт:** Учёт кредитов и депозитов.

**Что есть в API:** Кредиты и депозиты — это типы счетов (`accounts`), но нет специализированных эндпоинтов для:
- Начисления процентов
- Графика платежей по кредитам
- Капитализации по депозитам
- Напоминаний о платежах

**Что нужно добавить:**
- Эндпоинт `accounts.loan.schedule` — график платежей по кредиту.
- Эндпоинт `accounts.deposit.interest` — расчёт начисленных процентов.
- Параметры в `accounts.post/set`: `interest_rate`, `payment_day`, `end_date`, `loan_type`.

**Приоритет:** 🟡 Средний — базовая функциональность покрывается через `accounts`, но для полноценного учёта кредитов/депозитов нужны расширенные возможности.

---

### 3.11. Импорт/экспорт данных

**Что обещает сайт:** (Неявно — через EasyBank, но полезно и для ручного импорта.)

**Что есть в API:** Отсутствует.

**Что нужно добавить:**
- Эндпоинт `import.csv` — загрузка CSV-файла выписки.
- Эндпоинт `import.qif` — загрузка QIF-файла.
- Эндпоинт `export.csv` — выгрузка операций в CSV.
- Эндпоинт `export.xlsx` — выгрузка в Excel.

**Приоритет:** 🟢 Низкий — полезно, но не критично для MVP.

## 4. Приоритизированный список запросов к разработчику API

### 🔴 Высокий приоритет

| # | Запрос | Обоснование |
|---|--------|-------------|
| **H1** | **Самостоятельные эндпоинты `budget.post`, `budget.set`, `budget.categories.get`** | Без детализации по категориям невозможно реализовать бюджет, тахометр и аналитику. Текущий `budget.get` (только агрегат) непригоден для использования. |
| **H2** | **Поддержка повторяющихся операций (`operations.recurring.*`)** | Календарь плановых платежей — одна из ключевых фич сайта. Без рекуррентных операций пользователю придётся вводить каждый платёж вручную. |
| **H3** | **Гарантированная работоспособность `categories.post` / `categories.set`** | Кастомные категории — базовая потребность. Если эндпоинты не работают для сторонних приложений, это блокирует развитие. |

### 🟡 Средний приоритет

| # | Запрос | Обоснование |
|---|--------|-------------|
| **M1** | **Эндпоинты `tags.post` / `tags.set` / `tags.delete`** | Управление тегами невозможно без записи. Теги — важный инструмент классификации. |
| **M2** | **Эндпоинты аналитики (`analytics.summary`, `analytics.category`)** | Серверная аналитика снижает нагрузку на клиент и позволяет делать сложные вычисления. |
| **M3** | **Расширение `accounts` для кредитов/депозитов** (процентные ставки, график платежей) | Базовый функционал для пользователей с кредитами/депозитами. |
| **M4** | **Эндпоинт `goals.post` / `goals.set` (standalone)** | Текущая реализация через `users.post` неудобна и нестандартна. |
| **M5** | **Эндпоинты уведомлений (`notifications.*`)** | Напоминания критичны для вовлечения пользователей. |

### 🟢 Низкий приоритет

| # | Запрос | Обоснование |
|---|--------|-------------|
| **L1** | **Интеграция с банковскими агрегаторами (EasyBank)** | Требует серьёзных инвестиций и партнёрств; может быть реализована позже. |
| **L2** | **Эндпоинты импорта/экспорта файлов** | Полезно для power-users, но не критично для основной аудитории. |
| **L3** | **Эндпоинт рекомендаций** | Может работать на клиенте; серверные рекомендации — дополнительная ценность. |

## 5. Итоговая рекомендация

### Текущее состояние

API EasyFinance покрывает **базовые CRUD-операции** (счета, операции, теги, валюты, MCC-коды), но **существенно отстаёт** от обещаний, заявленных на сайте. Из 17 функций, представленных на homepage:

- **Полностью поддерживаются:** 2 (мультивалютность, MCC-коды/имя продавца)
- **Частично поддерживаются:** 6 (шаблоны, теги, категории, цели, бюджет, кредиты/депозиты)
- **Не поддерживаются:** 9 (рекомендации, календарь, EasyBank, аналитика, тахометр, уведомления, импорт/экспорт, повторяющиеся операции, прогнозы)

### Рекомендация по действиям

1. **Немедленно (blokеры):** Запросить H1 (бюджет по категориям), H2 (рекуррентные операции), H3 (категории для сторонних приложений). Без этих трёх пунктов Flutter-приложение не может реализовать обещанный функционал.

2. **В ближайшие 1–2 спринта:** Реализовать M1 (теги на запись), M2 (аналитика), M4 (standalone цели).

3. **Среднесрочно (1–3 месяца):** M3 (кредиты/депозиты расширенные), M5 (уведомления).

4. **Долгосрочно:** L1 (банковская синхронизация), L2 (импорт/экспорт), L3 (рекомендации на сервере).

### Приоритет для Flutter-приложения

Пока API не будет доработан, приложение может компенсировать недостающие функции **локальной бизнес-логикой**:

| Функция | Локальная реализация | Ограничения |
|---------|---------------------|-------------|
| Бюджет по категориям | Хранение в SharedPreferences/Hive | Данные не синхронизируются между устройствами |
| Повторяющиеся операции | Генерация при открытии календаря | Нет серверной валидации, нет push-напоминаний |
| Уведомления | Локальные push (flutter_local_notifications) | Только при открытом приложении, без SMS/email |
| Аналитика | Расчёт на клиенте из operations.get | Медленно при большом объёме данных |
| Теги (создание) | Локальное хранилище + маппинг на серверные ID | Дублирование данных, потеря при переустановке |

### Заключение

**Основной GAP** между сайтом и API составляет **≈60% функционала**. Flutter-приложение старается компенсировать это за счёт локальной логики, но это временное решение. Для полноценного соответствия обещаниям сайта необходима **целенаправленная доработка API** по приоритетам, описанным в разделе 4.

Рекомендуется провести совещание с разработчиком API для согласования roadmap и установления сроков по приоритетам H1–H3.
